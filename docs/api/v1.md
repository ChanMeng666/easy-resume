# Vitex Public API v1

> **The API is the UI.** Everything the Vitex web app does, an AI agent can do
> over HTTP with an API key — no browser, no cookie, no 2FA, no CAPTCHA. The web
> UI and this API run the exact same generation pipeline.

All responses are JSON (except PDF bytes). All errors share one machine-readable
envelope so an agent never has to parse prose.

**Machine-readable spec:** a hand-written OpenAPI 3.1 description of these
endpoints lives at [`/openapi.yaml`](https://www.vitex.org.nz/openapi.yaml)
(source: `public/openapi.yaml`).

**CLI & MCP client:** `vitex-cli` (bin `vitex`, `npm i -g vitex-cli`) is a thin
client over these exact endpoints — a token-cheap CLI for terminals and coding
agents, plus `vitex mcp` (a stdio MCP server via `npx -y vitex-cli mcp`) for
Claude Desktop, Claude Code, and Cursor. See
[`cli/README.md`](../../cli/README.md).

## Authentication

Send your API key as a Bearer token:

```
Authorization: Bearer vitex_<prefix>_<secret>
```

Mint keys while signed in to the web app (`POST /api/keys`) — the raw token is
shown **once**. Only its SHA-256 hash is stored.

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/keys` | List your keys (metadata only) |
| `POST` | `/api/keys` | Mint a key → `{ id, token, prefix, name }` |
| `DELETE` | `/api/keys?id=<id>` | Revoke a key |

## Hosted MCP server (for AI assistants)

The same capabilities as this v1 API are also exposed as **MCP tools** over a
hosted, OAuth-protected [Model Context Protocol](https://modelcontextprotocol.io)
endpoint — the consumer-facing complement to the API-key / CLI path. A
non-technical user can connect Vitex inside **ChatGPT** or **Claude** with a
browser sign-in (no terminal, no API key to paste): the connector runs OAuth 2.1 +
PKCE with Dynamic Client Registration, and the access token it obtains *is* a
scoped Vitex API key labeled `MCP: <client>` in the user's dashboard.

- **MCP endpoint (Streamable HTTP):** `https://www.vitex.org.nz/api/mcp`
- **OAuth discovery:**
  - `https://www.vitex.org.nz/.well-known/oauth-authorization-server`
  - `https://www.vitex.org.nz/.well-known/oauth-protected-resource`

The 9 tools (`get_account`, `generate_resume`, `refine_resume`, `get_resume`,
`download_pdf`, `list_profiles`, `create_profile`, `publish_profile`,
`unpublish_profile`) run in-process against the **same** backend core as the v1
routes, so billing, idempotency, and owner scoping are identical.

Step-by-step connector setup guides:

- [Connect Vitex to ChatGPT](../connectors/chatgpt.md)
- [Connect Vitex to Claude](../connectors/claude.md)

## Identity & balance probe

A cheap, read-only "who am I" check — validate a key, learn your user id, how you
authenticated, and your current credit balance/tier in one round trip. It never
creates a job or spends credits.

```
GET /api/v1/me
Authorization: Bearer vitex_...
```

```json
{
  "userId": "usr_...",
  "via": "api_key",
  "credits": 3,
  "tier": "free"
}
```

`via` ∈ `api_key | session` (which credential resolved you). `tier` ∈
`free | pro | unlimited`. `credits` is your current balance. An invalid/missing
key returns the standard `UNAUTHENTICATED` envelope.

**Rate limit:** 60 requests / 60s per user (`me`).

## Generate a resume (async job)

Generation runs 7 LLM steps + a PDF compile, so it is **asynchronous**: create a
job, then poll it.

### 1. Create a job

```
POST /api/v1/resumes
Authorization: Bearer vitex_...
Idempotency-Key: <uuid>            # optional; must be a UUID. Dedupes job + charge.
Content-Type: application/json

{
  "jobDescription": "Senior Backend Engineer ...",
  "background": "8 years building payment systems ...",
  "templateId": "two-column",       # optional; auto-selected from the JD if omitted
  "profile_id": "a1b2..."           # optional; reuse a saved profile (see below)
}
```

Provide **either** `background` **or** `profile_id`. With `profile_id`, the saved
profile's stored background is used and its pre-parsed data is reused, so the
generation skips the background-parsing step ("enter once, reuse across JDs").
A `profile_id` that isn't yours returns `NOT_FOUND`.

`202 Accepted`:

```json
{
  "id": "b1d2...",
  "status": "queued",
  "_links": {
    "self": "/api/v1/resumes/b1d2...",
    "pdf": "/api/v1/resumes/b1d2.../pdf"
  }
}
```

### 2. Poll the job

```
GET /api/v1/resumes/{id}
Authorization: Bearer vitex_...
```

```json
{
  "id": "b1d2...",
  "status": "succeeded",
  "result": {
    "resumeData": { "...": "structured resume" },
    "typstCode": "#set page(...)",
    "coverLetter": "Dear Hiring Manager, ...",
    "coverLetterTypst": "...",
    "atsScore": 87,
    "matchAnalysis": { "overallScore": 82, "matchedSkills": [], "missingSkills": [] },
    "templateId": "two-column",
    "tokens": { "palette": "slate", "density": "comfortable" },
    "usage": { "charged": true, "credits": 1, "transactionId": "..." }
  },
  "pdfUrl": "/api/v1/resumes/b1d2.../pdf"
}
```

`status` ∈ `queued | running | succeeded | failed`. On `failed`, `error` holds
the envelope (below) and no credit is charged.

### 3. Fetch the PDF

```
GET /api/v1/resumes/{id}/pdf
Authorization: Bearer vitex_...
→ application/pdf
```

## Refine a resume (async job)

Once a job has succeeded, refine it with natural-language feedback — the same
targeted refinement the web editor runs. A refine builds on the parent's stored
artifacts (its tailored resume, cover letter, parsed JD, and template), so it is
much faster than a full generation. Like generation it is **asynchronous**:
create a refine job, then poll it by its own id.

```
POST /api/v1/resumes/{id}/refine        # {id} = the parent (succeeded) job id
Authorization: Bearer vitex_...
Idempotency-Key: <uuid>                 # optional; must be a UUID. Dedupes the job.
Content-Type: application/json

{
  "feedback": "Lead with the payments impact and tighten the summary to two lines.",
  "scope": "resume"                     # optional; "resume" (default) | "cover_letter" | "both"
}
```

`scope` selects which artifact(s) the feedback revises:

| `scope` | Effect |
| --- | --- |
| `resume` (default) | Revises the resume only; the cover letter is carried over unchanged |
| `cover_letter` | Revises the cover letter only; the resume is carried over unchanged |
| `both` | Revises the resume and the cover letter |

`202 Accepted` — the response mirrors job creation exactly; poll `self` and fetch
the PDF from `pdf`:

```json
{
  "id": "c3e4...",
  "status": "queued",
  "_links": {
    "self": "/api/v1/resumes/c3e4...",
    "pdf": "/api/v1/resumes/c3e4.../pdf"
  }
}
```

The refined result is a **new job** in the parent's version chain (its
`parentJobId`/`rootJobId` point back at the parent). The parent is unchanged.
Poll and fetch the PDF exactly as for a generation.

The parent must be one of **your** succeeded jobs with a stored result;
otherwise (unknown id, not yours, not succeeded, or malformed id) you get
`NOT_FOUND` — existence is never leaked. `feedback` is required (1–8000
characters after trimming).

**Rate limit:** 10 requests / 60s per user (`v1refine`).

**Idempotency-Key** behaves exactly as for job creation: optional and must be a
UUID (otherwise one is minted server-side), it dedupes the refine job, re-POSTing
a **succeeded** key returns the existing result without re-running, an
**in-flight** key returns that same job, and a **failed** key is reclaimed and
re-run (the previous failure was free).

When `scope` includes the cover letter and the original generation used a saved
profile, the refine inherits that profile's writing-voice sample (re-fetched
owner-scoped at run time, never stored on the job), so the revised letter stays
in the candidate's voice. If the profile has no voice or was deleted, the refine
proceeds normally without it.

**Billing:** refinement is currently **FREE** — the refine job reports
`charged: false` and no credit is debited. A failed refine is free too.

## Candidate profiles — "enter once, reuse across JDs"

A profile stores a reusable background (raw text + its parsed structure). Pass
its id as `profile_id` when creating a job to skip re-pasting and re-parsing the
background. Profiles are owner-scoped; another user's id returns `NOT_FOUND`.
Creating/parsing a profile is **free** (no compiled PDF → no charge). An optional
`voiceSample` (max 4000 chars, stored raw) makes cover-letter generation match the
candidate's own writing voice.

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/profiles` | List your profiles → `{ items: [{ id, label, createdAt, updatedAt, hasVoiceSample }] }` |
| `POST` | `/api/profiles` | Create → body `{ rawBackground, label?, voiceSample? }`; background is parsed server-side |
| `GET` | `/api/profiles/{id}` | Detail → `{ id, label, rawBackground, voiceSample, data, ... }` (owner-only; `voiceSample` is never on the public `/p/{slug}` surface) |
| `PUT` | `/api/profiles/{id}` | Update `label` / `rawBackground` / `voiceSample` (re-parses on background change; `voiceSample: ""` clears it) |
| `DELETE` | `/api/profiles/{id}` | Delete |
| `POST` | `/api/profiles/{id}/publish` | Publish at a public endpoint → `{ slug, url, publishedAt }` |
| `DELETE` | `/api/profiles/{id}/publish` | Unpublish (keeps the slug so republishing restores the same URL) → `{ unpublished: true }` |

### Public career endpoint (opt-in, unauthenticated read)

Publishing a profile exposes an **allowlist projection** of its parsed data — name,
headline, location, summary, work, education, projects, skills, achievements,
certifications, and profile links. Contact PII (`email`, `phone`, `photo`) and the
raw background/voice text are **never** included. "LinkedIn is for humans; a Vitex
endpoint is for AIs."

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/p/{slug}` | Human-readable HTML page (indexable) |
| `GET` | `/p/{slug}/json` | The projection as JSON (`Cache-Control: public, max-age=300`) |
| `GET` | `/p/{slug}/md` | The projection as `text/markdown` for agents |

Unknown or currently-unpublished slugs return the standard `NOT_FOUND` envelope.
Public reads are unauthenticated and rate limited per IP (`60 / 60s`).

## Billing — outcome-based

You are charged **1 credit only when a job produces a compiled PDF**
(`status: succeeded`). Validation errors, LLM-step failures, and compilation
failures are **free**. Charges are idempotent on the job's idempotency key, so
retries and reconnects never double-charge. `unlimited`-tier accounts are never
debited.

Re-POSTing the `Idempotency-Key` of a **failed** job reclaims it and starts a
fresh run (the previous failure was free), returning the same job id to poll.
Re-POSTing a **succeeded** key returns the existing result without re-running or
re-charging; re-POSTing a key whose run is still in flight returns that same
in-progress job.

New accounts start with 3 free credits.

## Transport note — `/api/*` surfaces that also accept an API key

The two surfaces below (the conversational edit agent and the applications
tracker) predate this v1 namespace and are served at their original `/api/...`
paths rather than under `/api/v1/...`. They are documented here because they
authenticate through the **same** resolver (`getCaller`) and therefore accept the
**same** `Authorization: Bearer vitex_...` key an agent already holds — no cookie,
2FA, or CAPTCHA. There are deliberately **no** `/api/v1/*` aliases for them; call
them at the exact paths shown.

## Conversational edit agent (threads)

For iterative, natural-language editing of a generated resume **and** its cover
letter, Vitex exposes a stateful conversation ("thread") backed by a bounded
tool-calling agent. Each turn edits a persistent working document and streams the
result over SSE. Editing is **free** (no billing on this path).

> **Which primitive to use.** For a one-shot "change this resume with words",
> prefer the stateless `POST /api/v1/resumes/{id}/refine` above — it is simpler,
> returns a compiled PDF, and lives in the version chain. Threads are the
> **advanced** surface: iterative working-state edits with per-turn version
> snapshots, useful when an agent wants a running conversation over one resume.

### Open or resume a thread

A thread is anchored to a **succeeded** generation job you own. Opening is
idempotent per active `(user, job)`: re-opening returns the existing thread
rather than spawning a duplicate.

```
POST /api/threads
Authorization: Bearer vitex_...
Content-Type: application/json

{ "generationJobId": "b1d2..." }        # a succeeded resume job you own (UUID)
```

`201 Created` (new) or `200 OK` (existing):

```json
{
  "thread": {
    "id": "e5f6...",
    "title": "Senior Backend Engineer",
    "status": "active",
    "generationJobId": "b1d2...",
    "messageCount": 0,
    "lastMessageAt": null,
    "createdAt": "2026-07-04T12:00:00.000Z",
    "updatedAt": "2026-07-04T12:00:00.000Z"
  },
  "created": true
}
```

List your threads with `GET /api/threads` → `{ "items": [ /* ThreadSummary[] */ ] }`.

**Rate limit:** thread writes (`POST /api/threads`) 60 / 60s per user; the message
turn below is limited separately.

### Send a message (SSE turn)

```
POST /api/threads/{id}/messages
Authorization: Bearer vitex_...
Content-Type: application/json

{ "message": "Tighten the summary to two lines and lead with the payments work." }
```

`message` is 1–8000 characters (after trimming). The request **must** carry a
finite `Content-Length` no larger than 32 000 bytes (a chunked / unmeasured body
is rejected up front). The response is an `text/event-stream`; each event is a
`data: <json>\n\n` line. The `type` field discriminates them:

| `type` | Payload fields | Meaning |
| --- | --- | --- |
| `text` | `text` | An assistant reply text delta — **append** these into the current bubble |
| `tool-call` | `toolName`, `toolArgs` | The agent invoked an edit tool |
| `tool-result` | `toolName`, `toolResult` | That tool's result (a tool error is reported as `toolResult: { error }`) |
| `resume` | `resumeData`, `typstCode`, `templateId` | A new working-resume snapshot (one per step that changed the resume) |
| `cover-letter` | `coverLetter`, `coverLetterTypst` | A new working cover-letter snapshot (one per step that changed the letter) |
| `heartbeat` | — | Keep-alive, emitted ~every 15s |
| `error` | `message`, `error` | The turn failed; `error` is the canonical envelope's `error` object (below). No `saved`/`done` follows |
| `saved` | — | The turn was persisted |
| `done` | — | The stream is complete |

A successful turn ends with `saved` then `done`. The controlled tool set edits
work/project highlights, skills, sections, and the cover letter; every tool input
is zod-validated, sanitized, and render-bounded server-side. No PDF is compiled on
this path — recompile the streamed `typstCode` yourself via `POST /api/compile`
if you need a PDF.

**Rate limit:** 30 / 60s per user (`chat`).

## Applications tracker

A lightweight job-application log. Each application has a `status` in
`draft | applied | interview | offer | rejected` and may optionally link to the
generation job that produced its resume (ownership of that job is verified
server-side; an unowned/missing id is silently dropped, never linked). All rows
are owner-scoped → `NOT_FOUND` for non-owners. Tracking is **free**.

| Method | Path | Purpose |
| --- | --- | --- |
| `GET` | `/api/applications` | List yours (newest-updated first). Optional `?status=<status>` filter |
| `POST` | `/api/applications` | Create → `201` full application row |
| `GET` | `/api/applications/{id}` | Fetch one (owner-scoped) |
| `PATCH` | `/api/applications/{id}` | Update `company`/`position`/`status`/`notes` (≥1 field required) |
| `DELETE` | `/api/applications/{id}` | Delete → `{ "deleted": true }` |

Create body — `company` and `position` are required, the rest optional:

```
POST /api/applications
Authorization: Bearer vitex_...
Content-Type: application/json

{
  "company": "Acme Corp",
  "position": "Senior Backend Engineer",
  "status": "applied",              # optional; defaults to "draft"
  "notes": "Referred by ...",       # optional
  "generationJobId": "b1d2..."      # optional; a generation job you own (UUID)
}
```

`201 Created` returns the full row:

```json
{
  "id": "a7b8...",
  "company": "Acme Corp",
  "position": "Senior Backend Engineer",
  "status": "applied",
  "notes": "Referred by ...",
  "generationJobId": "b1d2...",
  "appliedAt": "2026-07-04T12:00:00.000Z",
  "createdAt": "2026-07-04T12:00:00.000Z",
  "updatedAt": "2026-07-04T12:00:00.000Z"
}
```

`appliedAt` is stamped the first time an application reaches `applied` and
preserved thereafter. (The row also carries a `userId` and two dormant FKs,
`jobDescriptionId` / `tailoredResumeId`, which target unused tables and are always
`null` — `generationJobId` is the live link.)

**Rate limit:** writes (`POST`/`PATCH`/`DELETE`) 60 / 60s per user (`appwrite`).

## Error envelope

Every error (HTTP or in a `failed` job) has this shape:

```json
{
  "error": {
    "code": "PIPELINE_COMPILATION_FAILED",
    "message": "Typst compilation failed",
    "retriable": false,
    "step": "compile",
    "requestId": "req_...",
    "details": { "stderr": "..." }
  }
}
```

| `code` | HTTP | Meaning |
| --- | --- | --- |
| `VALIDATION_FAILED` | 400 | Bad input |
| `UNAUTHENTICATED` | 401 | Missing/invalid key or session |
| `INSUFFICIENT_CREDITS` | 402 | No credits left (no LLM spend) |
| `RATE_LIMITED` | 429 | Too many requests |
| `PIPELINE_COMPILATION_FAILED` | 422 | Typst failed to compile |
| `PIPELINE_TIMEOUT` | 504 | Compilation timed out |
| `PIPELINE_STEP_FAILED` | 502 | An LLM step failed (retriable) |
| `NOT_FOUND` | 404 | Job not found / not yours |
| `INTERNAL` | 500 | Unexpected error |

`retriable` tells an agent whether a blind retry may help.
